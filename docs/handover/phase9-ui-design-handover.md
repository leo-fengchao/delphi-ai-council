# 集思 · Delphi — Phase 9 UI Design Handover

> 给 **Claude Design** 的交接文档。你**不需要读代码**，凭本文件即可产出 UI demo（静态 HTML/CSS 或 React + 假数据均可）。
> 沟通与产出文案：**只用简体中文**。

---

## 0. 一句话产品定位

「**多 AI 议会**」浏览器插件：用户问一个问题 → 插件驱动多家 AI 在各自原生网页作答 → 互相匿名评审打分 →（可选）主席多轮追问辩论 → 由一个「主席 AI」综合出最终答案，并给出**共识点 / 争议区 / 置信度**。全程零 API key、复用用户已登录的各家网页。

中文名「集思」（集思广益），英文名「Delphi」（德尔斐法：多专家多轮独立给意见再收敛成共识）。气质要求：**亲和、温暖、消费级**，不是极客工具感。

---

## 1. 这次设计的目标（Phase 9 要解决什么）

**现状痛点**：当前界面把所有 AI 的回复**全量平铺**成一张张大卡片，信息过载，普通用户看不懂也看不完。一场议会有 5~8 家 AI × 多个阶段（作答/评审/辩论/总结），全铺出来就是一面文字墙。

**本期目标**：从「文字墙」改成「**概览 + 按需详情**」。具体五件事：

1. **成员状态方块（核心）**：每个参会 AI 是一个**小方块/卡片**，显示该模型当前处于哪个阶段、一句话摘要，而不是全文。
2. **弹窗看详情**：点方块或某个阶段标签 → 弹窗里才展示该成员该阶段的**完整原文**。
3. **一键追问**：每个成员方块上有「追问」按钮，点了**跳转到该 AI 的原生网页**继续聊（插件**不做**站内追问）。
4. **历史议事侧栏**：左侧历史记录栏（对标主流 AI 产品的会话列表），点开可回看过去某场议会的各家回答与主席结论。
5. **整体视觉升级**：当前是最朴素的内联样式原型，这次要给出真正的设计语言（配色/间距/层级/状态色）。可评估引入轻量组件库（候选 **Hero UI**），但**保持克制、避免过度设计**。

> 注意：这是一个**浏览器插件的常驻扩展页**（一个独立的浏览器 Tab，全宽桌面端，不是手机、不是 popup 小弹窗）。按桌面 Web App 来设计。

## 2. 完整流程与「阶段」枚举（设计状态机时照此来）

一场议会按顺序经过这些**顶层阶段**（成员方块上要能体现成员当前在哪一阶段）：

| 阶段 | 名称 | 含义 | 是否每家都参与 |
|---|---|---|---|
| `stage1` | **初次作答** | N 家 AI 各自独立回答原始问题，互不可见 | 全部成员 |
| `stage2` | **交叉评审** | 各家回答被匿名为 Response A/B/C…，互相打分排序 | 全部成员 |
| `debate` | **多轮辩论**（可选，默认关） | 主席对匿名成员**定向追问**，最多 3 轮 | 仅被主席点名的成员 |
| `stage3` | **主席综合** | 主席 AI 汇总全部 → 输出最终答案 + 共识/争议/置信度 | 仅主席 |
| `finished` | **已完成** | 出结论 | — |

**单条成员（“腿”）的状态**：`pending`（待开始）/ `running`（进行中）/ `done`（完成）/ `failed`（失败）。
running 时还有细分小阶段文案：`注入问题…` / `已发送，生成中…` / `生成中…` / `抽取回答…`。

**失败态**要友好提示，常见几类：需要登录、出现人机验证、没找到输入框、没抓到回答、超时。失败的成员方块上要有「**重试**」按钮。

**降级原则**：任一家失败，议会用**剩余成员继续**，绝不整盘崩。置信度会据实际参与人数下调——UI 要能体现「X 家参与 / Y 家失败」。

---

## 3. 关键角色：主席（Chairperson）

- 用户在发起前从参会成员里**选一个**作主席（👑）。
- 主席在 stage3 负责综合，在 debate 里负责发问。
- 主席**强制开启深度思考**；其它成员的深度思考由总开关控制。
- UI 上主席要有明显标识（皇冠/高亮/徽章），在成员列表、辩论时间线、结论区都要点出「谁是主席」。

## 4. 需要设计的页面区块清单

按从上到下 / 从左到右组织，给设计师一个完整的画面：

### A. 顶部 / 品牌区
- 产品名「集思 · Delphi」+ slogan（可用「汇聚众智，收敛共识」之类）。
- 当前议会状态指示（idle / 进行中 / 已完成）。

### B. 左侧：历史议事侧栏（新增）
- 类似 ChatGPT/Claude 的会话列表：每条 = 一场历史议会，显示**问题摘要 + 时间 + 参会家数**。
- 点击加载回看该场的各家回答与主席结论（只读）。
- 顶部有「新议事」按钮。
- 需要考虑**折叠/展开**（毕竟主区域要给成员方块留空间）。

### C. 输入与配置区
- **问题输入框**（多行 textarea）。
- **参会成员选择**：8 家 AI 用 chip/标签多选 —— DeepSeek、Kimi、通义千问、豆包、腾讯元宝、ChatGPT、Claude、Gemini。选中态要清晰。
- **主席选择**：下拉，只能从已选成员里选，带 👑。
- **两个开关**：
  - 「深度思考」（默认**开**）——议事质量基线。
  - 「多轮辩论」（默认**关**）——更慢更耗，高价值问题再开。
- **主操作按钮**：「发起议事（N 家）」。进行中变为「议事中…」+ 出现「⏹ 中止议事」「🧹 关闭所有 AI 窗口」。

### D. 成员状态方块区（本期核心）
- 每个参会 AI 一个方块。方块内：
  - AI 名称 + 图标（主席带 👑）。
  - **当前阶段徽章**（初次作答 / 交叉评审 / 辩论第 N 轮 / 主席综合 / 已完成 / 失败）。
  - **一句话摘要**（回复的前一两句，不是全文）。
  - 进行中有 loading/进度感；失败有错误色 + 「重试」。
  - 「**查看详情**」（弹窗）与「**追问**」（跳原生页）两个动作。
- 布局建议网格（grid），5~8 个方块要排得整齐、不拥挤。

### E. 详情弹窗（Modal）
- 点方块或某阶段 → 弹窗展示**该成员该阶段的完整原文**（长文，要可滚动）。
- 最好能在弹窗内切换看该成员的**不同阶段**（初次作答 / 它写的评审 / 辩论回复）。

### F. 多轮辩论时间线（debate 开启时才显示）
- 一条**时间线**：每轮里「👑 主席 → Response X：追问内容」+「Response X：回复内容」。
- 多轮纵向累积，能看出第 1/2/3 轮。
- 注意：辩论里成员是**匿名编号**（Response A/B/C），主席看不到真实身份；但 UI 给用户看时**可以**揭示「Response A 其实是 Kimi」这类映射（中枢持有映射）——这点可由你决定是否揭示，给两种方案更好。

### G. 主席综合结论区（最重要的产出，要最显眼）
- 「👑 主席综合结论（某模型）」。
- 四块内容：
  - **最终回答**（主体，最大篇幅）。
  - ✅ **共识点**。
  - ⚠️ **争议区**。
  - **置信度**：0~100 的分数，要有可视化（环形/进度条/大数字），并能体现「基于 N 家参与」。
- 这一区是用户真正要的答案，视觉权重应高于成员方块。

### H. 恢复提示条
- 插件页刷新/崩溃后能检测到「上次未完成的议会」，顶部出一条提示：「检测到未完成议会…是否续跑？[恢复][丢弃]」。给个样式即可。

### I. 适配校准区（已有功能，低优先级，给个收纳样式即可）
- 一个可折叠面板，列出各站点的「校准/贡献/重置」按钮。**不是本期重点**，能优雅折叠收起不干扰主流程就行。

## 6. 设计约束与品味

- **桌面 Web App**（独立 Tab，全宽）。中文界面为主。
- 气质：亲和、温暖、消费级；**不要**极客黑客风。
- **克制**：呼应技术栈原则「React 轻量用法，避免过度设计」。组件库 Hero UI 可选可不选——给方案即可，最终由产品方定。
- 状态色建议成体系：进行中（蓝）/ 成功（绿）/ 警告争议（黄）/ 失败（红）/ 主席（金）。
- 暗色模式：可选加分项，不强制。
- 长文很多（AI 回复动辄上千字），**信息密度管理**是设计成败关键——默认收起、按需展开是核心思路。

## 7. 交付物期望

1. **整体视觉风格**：配色、字体、间距、圆角、阴影的一套设计语言。
2. **主界面 demo**：覆盖「空闲态 / 议事进行中 / 出结论」三个关键状态。
3. **成员方块**的几种状态（pending/running/done/failed/主席）。
4. **详情弹窗**、**辩论时间线**、**结论区（含置信度可视化）**、**历史侧栏**。
5. 形式不限：静态 HTML/CSS 或 React + mock data 都行，能直观看到效果即可。
