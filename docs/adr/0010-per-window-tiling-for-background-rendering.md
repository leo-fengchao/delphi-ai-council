# ADR-0010: 每站一窗、并排平铺以解决后台标签页不渲染

- **日期:** 2026-06-03
- **状态:** Accepted（修订 ADR-0004 的标签页承载方式；标注为「先确保功能」的基线实现）

## 上下文 (Context)
实测发现：把 N 个 AI 站点放进后台/非激活标签页时，注入、发送、抽取**全部不可靠**，且各站表现不一致。根因不是选择器，而是浏览器对**隐藏/非激活标签页**的限制：

- `requestAnimationFrame` 被强制暂停 → 依赖 rAF 的流式渲染停摆；
- `element.focus()` 无法真正获得焦点 → contenteditable 的 `paste`/execCommand 注入落空（如 Kimi 输入框为空）；
- 合成的鼠标点击不触发 → 点按钮发送失效（如豆包有内容但发不出）；
- 虚拟列表（DeepSeek/豆包）不渲染消息行 → 抽取为空，切到前台才出现。

`visibility.content.ts` 的「可见性伪装」只能骗过站点**自身**基于 `document.hidden` 的暂停逻辑，**无法**消除浏览器内核层面对隐藏标签页的 rAF/焦点限制。

## 决策 (Decision)
采用 **每站一窗、并排平铺** 的执行模式（修订 ADR-0004 原「独立窗口装 N 个标签页 / 当前窗口标签组」的方案）：

- 为每个参会站点单独 `chrome.windows.create` 一个窗口，按屏幕可用区域切成网格平铺（`council-tabs.ts` 的 `tileGrid`）。
- 窗口以 `focused:false` 创建：**可见、可渲染**（每个窗口的唯一标签页即激活标签页），但不抢占用户正在看的 Council Page 焦点。
- 平铺保证窗口互不完全遮挡 → 规避 macOS 窗口遮挡（occlusion）导致的隐藏 → 各站真正渲染。
- 配合 `visibility.content.ts`（可见性伪装）与 `autoDiscardable=false`（防丢弃）。

## 后果 (Consequences)
**好处：**
- 注入/发送/抽取在「激活且可见」的标签页中稳定工作，跨站一致。
- 实现简单直接，先把多站点议会功能跑通。

**代价 / 妥协：**
- **占屏**：站点多时平铺会铺满屏幕、盖住 Council Page（议会期间用户需从任务栏/Dock 切回看结果）。
- 并非长期形态。**优化项（进 roadmap）**：探索更省屏的「渲染保活」——如最小化但保持渲染、离屏渲染、或「后台生成 + 仅抽取瞬间短暂激活」的混合策略，在可靠性与占屏间取得更好平衡。
- 与 ADR-0004「独立窗口」精神一致，但承载粒度从「一窗多标签」改为「一站一窗」。
