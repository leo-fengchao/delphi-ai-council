# ADR-0001: 全程驱动原生网页、零 API 的核心路线

- **日期:** 2026-06-02
- **状态:** Accepted

## 上下文 (Context)
要做一个"多 AI 议会"插件。两种实现路线：
1. **API 路线**（AI Council Mac App 的做法）：调用各家 API，I/O 干净、可结构化、编排可靠，但要 API key、有费用，且原生联网搜索/原生 system prompt 的完整能力可能无法等价复现。
2. **原生网页路线**（AIChatIndex 的机制范式）：靠 DOM 自动化驱动各家原生网页，复用用户登录态、零 key、零费用、保留各家最完整的原生联网与系统提示。

产品方明确要求：**全程走原生网页，绝不调用任何 API**（硬约束 C1）。

## 决策 (Decision)
采用**纯原生网页路线**：
- 所有"发送 prompt / 接收回答"均通过 content script 的 DOM 操作完成。
- 议会三阶段（作答、评审、主席综合）全部在原生网页间编排，**无 API 兜底**。
- 不持有、不传输任何 API key。

## 后果 (Consequences)
**好处：** 零费用、零 key、复用登录态、保留各家原生联网搜索与优化过的 system prompt——即本产品相对 API 类竞品的核心差异化。

**代价 / 妥协：**
1. **脆弱性**：依赖各站点 DOM 结构，站点改版即失效 → 由 ADR-0005（远程热更新配置）缓解。
2. **无兜底**：编排任一环节失败无法切 API → 必须在 ADR-0006 设计强降级。
3. **⚠️ 存在性风险（无法用工程消除）**：
   - 自动化驱动 ChatGPT/Gemini 等网页 UI 违反各家 ToS；Gemini 与 Chrome 商店同属 Google。
   - "广播 + 展示"有商店先例（AIChatIndex/ChatHub 在架），大概率能过审；但"自动抓取 + 自动回灌做交叉评审"更像滥用式自动化，**真实风险是上线起量后被 provider 投诉下架**。
   - 此风险已向产品方明示并由其接受。缓解方向（半自动确认、平台风险不对称）见 README Q1 / 待建 ADR。
4. 各家可能触发 bot 检测，导致**用户账号**被标记——属产品质量与信任问题，需在降级与提示上处理。
