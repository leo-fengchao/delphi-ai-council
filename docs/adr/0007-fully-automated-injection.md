# ADR-0007: 评审/总结阶段采用「全自动注入发送」

- **日期:** 2026-06-02
- **状态:** Accepted（取代 ADR-0006 中关于 Q1 的待定项）

## 上下文 (Context)
ADR-0006 第 44 行将「阶段二/三是否需要用户一键点发送的半自动形态」留作 Q1 待产品方确认。该决策同时牵动两件事：
1. **合规与下架风险**：在 ADR-0001 的零 API 约束下，所有发送都靠 DOM 自动化点击；全自动会被部分 provider 判为滥用式自动化，是「上线后被投诉下架」风险的主要放大器。
2. **状态机形态**：半自动需要在每次发送前插入「等待用户确认」的人工节点；全自动则是连续推进的纯机器流。

产品方已在 2026-06-02 明确选择 **全自动注入发送**，优先体验顺滑度。

## 决策 (Decision)
- 三阶段议会全程**全自动**：注入 prompt 后由适配器直接触发各站点的发送动作，无人工确认节点。
- 适配器协议（ADR-0005）的 `input.method` 在「填入输入框」后**继续执行 submit 动作**（回车或点击发送按钮，按各站点配置）。
- XState 编排状态机（ADR-0004）**不引入** `awaitingUserConfirm` 等待态；每条腿为 `inject → submit → awaitCompletion → extract` 的自动序列。
- 为缓解 C1+C2 风险，作为补偿性约束：
  - 注入/发送之间加入**随机化的人类节奏延迟**（避免零间隔的机器特征），具体参数后续在适配器配置中调优。
  - 保留**全局急停**（用户一键中止整场议会）。
  - 不规避、不伪装；仅复用用户已登录态，遵循 ADR-0001 的既定立场。

## 后果 (Consequences)
**好处：**
- 体验最顺滑，一次输入全程无人值守，最贴近「消费级产品」定位（呼应品牌「集思」气质）。
- 状态机更简单，无人工等待态，崩溃恢复路径更短。

**代价 / 妥协：**
- **放大下架风险**：这是 ADR-0001 已记录的存在性风险中、最难用工程消除的一项。海外站点（C3）尤甚。
- 一旦某 provider 收紧自动化检测，可能需要**回退到半自动**——届时按本 ADR 的反向决策新建 ADR，并在状态机中加回确认节点。
- 全局急停与人类节奏延迟是必要的最低限度缓解，不能消除风险，只能降低触发概率。
