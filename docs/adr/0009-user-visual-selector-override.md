# ADR-0009: 用户可视化校准 + 本地选择器覆盖层（Phase 3）

- **日期:** 2026-06-03
- **状态:** Accepted（Phase 3 主线，优先于韧性层）

## 上下文 (Context)
ADR-0005 把适配器选择器数据驱动化、ADR-0008 让其可远程热更新，但二者仍依赖**开发者/维护者**先写对选择器。实测中（DeepSeek/Kimi/千问/豆包/元宝）反复出现：

- 站点改版导致选择器频繁失效；
- 不同站点输入控件差异极大（textarea / ProseMirror / Quill / Lexical / Slate），注入与发送方式各异；
- 维护者手调是单点瓶颈，且无法覆盖所有站点 × 所有版本。

结论：只要「适配能力」锁在维护者手里，产品就长期脆弱。需要把这份能力**下放给用户**。

## 决策 (Decision)
新增 **Phase 3：用户可视化校准 + 本地覆盖层**，优先于韧性层（XState/Watchdog 顺延 Phase 4）。

### 1. 页面内「拾取模式」
- 在 AI 原生页注入一个可开启的拾取覆盖层：高亮悬停元素，用户依次点选 **输入框 / 发送按钮 / 回答区 / 停止按钮 / 深度思考开关** 等角色。
- 每次点选生成**稳健选择器**：优先 `id` → `data-*`/稳定属性 → 角色属性 → 结构化短路径；避免使用易变的动态类名（如 Tailwind 原子类、CSS Modules 哈希后缀）。

### 2. 本地用户覆盖配置
- 用户校准结果以「站点 → 角色 → 选择器/策略」的形式存入 `chrome.storage.local`。
- **ConfigLoader 合并优先级（在 ADR-0008 三级回退之上再加一层）：用户覆盖 > 远程 > 内置兜底。**
- Council Page 提供「校准此站点」入口，以及覆盖项的查看 / 单项重置 / 全部清除。

### 3. schema 扩展
- 适配器 schema（ADR-0005）增加可选的可点选角色。
- **深度思考开启用有序步骤数组 `thinkingActivation: string[]`**（取代最初设想的单个 `thinkingToggle`）：
  - 单步站点长度为 1；多步站点按序点击，例：豆包先点入口按钮再点「思考」选项（2 步）、Kimi 先切模型再选「思考」模型（2 步）。
  - 运行时在「发送前」（仅当用户勾选深度思考）按序「等待元素出现 → 点击 → 略等」逐步执行；某步缺失则放弃后续但仍继续发送，不阻断主流程。
  - 校准录制：拾取器对这些步骤启用 `clickThrough`（拾取后真点击），使前一步的菜单/选项弹出，用户方能继续点选下一步。

### 4. 安全边界
- 覆盖配置**只存选择器与策略字符串，绝不存可执行代码**（延续 ADR-0005 安全约束）。
- 本阶段仅本地生效；**共享/众包是 Phase 6 的独立工作**，需配审核与信任机制，不在本 ADR 范围。

## 后果 (Consequences)
**好处：**
- 站点改版后用户可**自救**，不必等维护者发版或推配置 → 直接消解 ADR-0005 脆弱性的根因。
- 为 Phase 6 众包共创打基础（本地覆盖即可提交为社区资产，远程源即分发通道）。
- 把「深度思考开关」等交互纳入统一的可点选框架，而非逐个硬编码。

**代价 / 妥协：**
- 需要实现一套可靠的「点选 → 稳健选择器生成」算法（跨 Shadow DOM / iframe / 动态类名是难点）。
- 增加用户认知成本：需引导非技术用户理解「点选哪个元素」。
- 韧性层（Phase 4）相应顺延。
