# ADR-0004: 标签页生命周期韧性（autoDiscardable + 独立窗口 + 可恢复状态机 + 看门狗）

- **日期:** 2026-06-02
- **状态:** Accepted

## 上下文 (Context)
ADR-0002 选用真实标签页后，引入了标签页生命周期风险。产品方明确提出担忧：这些标签页可能被 **Chrome 内存节省器自动丢弃 (discard)**、被**用户误关**、或被**手动清理**，从而导致整个议会流程崩溃。

不能把流程设计成"假设标签页一直存在"。

## 决策 (Decision)
采用**"丢了能恢复"**的多层防护，而非"保证不丢"：

1. **抑制自动丢弃**：会话期间对所有议会标签页设置
   `chrome.tabs.update(tabId, { autoDiscardable: false })`，阻止 Chrome 内存节省器自动丢弃。（无法阻止操作系统级极端内存压力，属边缘情况。）
2. **集中在标签组（修订）+ 清晰标注**：议会标签页集中放在**用户当前窗口的一个标签组**里（`chrome.tabs.group` + `chrome.tabGroups`，组名「集思议会」），后台打开、不抢占焦点。
   - **修订原因（2026-06-02 测试反馈）**：原方案用「扩展自建的独立窗口」在后台打开，用户在最大化的 Council Page 后点「发起议事」后看不到任何动静、不知发生了什么。改为同窗口标签组后，标签页直接出现在当前窗口的标签栏，用户可一眼看到各 AI 网页已打开并随时点入观察。
   - 仅当某站点需要登录时，才把对应标签页切到前台引导（`foregroundTab`）。
3. **可恢复状态机**：议会全量状态（当前阶段、各腿是否完成、已抓取的输出、原始 prompt）持久化到 `chrome.storage.local`。每条腿（每个 AI 的一次交互）的 prompt 设计为**幂等**，可安全重发。
4. **Watchdog 看门狗**：在 Council Page 持续比对"应有状态 vs 实际标签页状态"：
   - 监听 `chrome.tabs.onRemoved`（关闭）与 `onUpdated` 的 `discarded` 标志（丢弃）。
   - 维护到各 content script 的长连接 port，`onDisconnect` 即视为该标签页失联。
   - 每条腿设超时 (`maxWaitMs`)，卡住即判失败。
   - 任一异常 → 重开标签页 → 等待 content script 与登录就绪 → 重发该腿 prompt → 重新抓取。**单个标签页的意外只表现为"重试一次"，不致整盘崩溃。**

## 后果 (Consequences)
**好处：** 标签页被丢弃/误关/卡死都可自动恢复；整盘议会可在 Council Page 重新打开后从持久化状态续跑。

**代价 / 妙协：**
- 恢复逻辑增加复杂度（重连、登录就绪检测、幂等重发）。
- 重发会让被重置的那家 AI **重新生成一次**，消耗用户该平台的额度/速率，需在 UI 透明提示。
- 登录态在重开后若失效（被登出），无法自动解决，须降级为提示用户手动登录该家。
