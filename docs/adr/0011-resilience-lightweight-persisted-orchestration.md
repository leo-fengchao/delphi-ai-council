# ADR-0011: 韧性层先用轻量持久化编排，XState 推迟到 Phase 5

- **日期:** 2026-06-03
- **状态:** Accepted（Phase 4 实现方式；不改变 ADR-0004 的韧性目标）

## 上下文 (Context)
ADR-0004 定下韧性目标（autoDiscardable + 可恢复状态机 + Watchdog + 幂等重发），README 技术栈表把 **XState** 列为「编排状态机」的推荐选型。进入 Phase 4 时面临实现方式抉择：

1. **现状**：Phase 1–3 已用 Promise 编排（`orchestrator.ts` 的 `broadcastStageOne` / `driveLeg`）逐行实机调通 8 家站点（含富文本注入、验证码、登录引导、深度思考），是经过验收的稳定主路。
2. **XState 重写**最贴合文档，但要把已验证的广播主路较大幅度重构，且本阶段无法由开发者自行实机测试，存在跑回归风险。
3. 韧性的**实际价值**落在「状态持久化 + Watchdog + 幂等重发 + 崩溃恢复」这几件事上——它们与状态机用何种库实现是正交的。

## 决策 (Decision)
**Phase 4 采用「轻量持久化编排」：在现有 Promise 主路之上叠加韧性能力，不引入 XState。**

- 用一个显式的 `SessionState` 类型（`shared/session-state.ts`）建模会话与各腿状态，持久化到 `chrome.storage.local`（单活动会话）。
- **Watchdog**（`council-page/watchdog.ts`）监听 `chrome.tabs.onRemoved` / `onUpdated(discarded)` / `chrome.windows.onRemoved`：被丢弃即重新设 `autoDiscardable=false`（重新保护）；被关闭即标记该腿「丢失」，供恢复。
- **幂等重发**：仅当某腿的标签页/窗口**确实消失**（基础设施失败、本就没产出答案）时自动重开并重发，且设次数上限；对「未登录 / 验证码 / 抽取为空 / 找不到输入框」等**内容级失败不自动重发**（避免无谓消耗用户平台额度，呼应 ADR-0004 的额度提示），保留既有的手动「重试」入口。
- **崩溃恢复**：Council Page 载入时检测持久化会话，提示「恢复/丢弃」；恢复时对未完成腿重开重发、已完成腿直接展示存档输出。

**XState 推迟到 Phase 5**：彼时要新增「匿名互评 + 主席综合」两个阶段，状态显著变复杂，引入状态机库的收益才划算；届时另立 ADR 说明迁移。

## 后果 (Consequences)
**好处：**
- 改动面小、不动已验证的广播主路，回归风险低，不增加依赖。
- 完整交付 ADR-0004 的韧性目标（恢复能力与 XState 方案等价）。

**代价 / 妥协：**
- 状态流转是手写的显式状态字段而非声明式状态机，阶段增多后可读性/防错性弱于 XState —— 这正是 Phase 5 迁移的动机。
- 自动重发限定在「标签页消失」一种情形，覆盖面有意保守（优先不浪费额度）；其余失败仍依赖用户手动重试。
