# ADR-0006: 三阶段议会编排的 DOM 实现与单点降级

- **日期:** 2026-06-02
- **状态:** Accepted（阶段二/三是否半自动取决于 README Q1，定后可能新增 ADR）

## 上下文 (Context)
AI Council 的灵魂是三阶段评议（并行作答 → 匿名交叉评审 → 主席综合）。在 ADR-0001 的零 API 约束下，这套编排只能落到 DOM 操作上，且无 API 兜底，必须自带强降级。

## 决策 (Decision)

### 阶段一 · 并行作答
- Orchestrator 向 N 个标签页**并行注入原始问题**（按各适配器的 `input.method`）。
- 各家在自己原生网页内作答，**互不可见**。
- 按 `completion` 规则检测每家完成 → 按 `extraction` 抽取干净回答（统一转 markdown）。

### 阶段二 · 匿名交叉评审
- **匿名化在插件侧完成**：把各家回答重标为 `Response A / B / C …`，隐去来源。
- 向评审员标签页注入评审 prompt（含全部匿名回答，要求排序/打分/指出问题）。
- 评审员是否为全员或抽样，见 Q4。
- 抽取各评审结果。

### 阶段三 · 主席综合
- 选定主席标签页（固定或用户选，见 Q3）。
- 注入：全部匿名回答 + 全部评审结果。
- **要求主席以固定分隔格式输出**，便于稳定解析，例如：
  ```
  ===ANSWER=== ... ===/ANSWER===
  ===CONSENSUS=== ... ===/CONSENSUS===
  ===DISPUTED=== ... ===/DISPUTED===
  ===CONFIDENCE=== 0-100 ===/CONFIDENCE===
  ```
- 抽取后按分隔符解析，渲染综合答案 / 共识点 / 争议区 / 置信度卡片。
- 置信度是否再叠加插件侧文本相似度校验，见 Q6。

### 单点降级（关键，因无 API 兜底）
- 任一站点失败/限流/超时/未登录：
  - **不整盘崩溃**；用剩余成员继续后续阶段。
  - 置信度按实际参与成员数下调，并在 UI 明示"X 家中 Y 家参与"。
  - 若主席所在家失败，按预设顺序改选下一家为主席。
  - 评审员不足时可跳过阶段二，直接进主席综合并标注"未经交叉评审"。

### 阶段间衔接
- 每阶段结束即把状态持久化（配合 ADR-0004），任一阶段可在崩溃恢复后续跑。
- 阶段二/三是否需要"用户一键点发送"的半自动形态，由 Q1 决定——这直接影响商店合规与状态机的人工等待节点。

## 后果 (Consequences)
**好处：** 在零 API 下完整复现议会三阶段；强降级保证可用性。

**代价 / 妥协：**
- 评审/主席阶段要把多家长回答拼进输入框，prompt 体量大，注入与抽取更易出错；依赖适配器配置（ADR-0005）持续校准。
- 固定格式解析依赖模型遵循格式，需做容错解析与重试。
- 全流程耗时显著高于 API 路线（多次串行的网页生成 + 等待），需用流式状态 UI 管理用户预期。
