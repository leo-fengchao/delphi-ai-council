# ADR-0012: 引入 XState 重构议会编排为声明式状态机

- **日期:** 2026-06-03
- **状态:** Accepted (Phase 5 核心架构)

## 上下文 (Context)
随着 Phase 5 完整三阶段议会（并行作答 → 匿名交叉评审 → 主席综合）的落地，系统面临的状态复杂度显著增加：
1. **多阶段流转**：从单次的“下发-提取”变为依赖前一阶段产出的多轮交互。
2. **状态共享与聚合**：阶段二需要阶段一的所有回答进行匿名化拼接，阶段三需要汇总前面所有产出。
3. **降级与韧性**：在多阶段长流程中，任一站点的网络错误、超时或未登录，需要精准控制是局部降级（不阻断整体进程）还是触发全局恢复。
4. **UI 绑定**：UI 需根据复杂状态（如 "正在抽样"、"正在综合"）给予用户即时反馈。

现有的基于 Promise 和显式状态字段（ADR-0011 临时轻量编排）的实现在应对这种多分支、多阶段生命周期时会导致严重的面条代码，极难维护。

## 决策 (Decision)
**采用 XState (v5) 将 Council Orchestrator 重构为标准的声明式状态机 (State Machine)，并以 `@xstate/react` 接入 UI。**

具体设计原则：
1. **统一 Context (上下文)**：将 `SessionState` 中的各种信息（当前轮次结果、选定的主席模型、匿名化映射等）全部置于 XState 的 `context` 中，随状态流转不可变地更新。
2. **顶层状态划分**：
   - `idle`: 初始状态，等待用户提交问题。
   - `stage1_answering`: 阶段一，广播问题给全体 AI，并行收集回答。
   - `stage2_reviewing`: 阶段二，根据 Q4 决策，实施**全员互评**。
   - `stage3_summarizing`: 阶段三，选定主席（根据 Q3 决策，**用户可选**，若失败可顺延），注入所有匿名回答和互评结论要求输出。
   - `done`: 议会圆满/降级结束，输出结论。
   - `error`: 全盘性不可逆错误（极少触发，大部分情况走降级完成）。
3. **结合现有韧性设施**：
   - 继续沿用 `Watchdog` 和 `chrome.storage.local` 进行持久化（在 XState 的状态迁移 actions 中写入存储）。
   - 保留 `council-tabs` 和并排平铺逻辑。

## 后果 (Consequences)
**好处：**
- 状态流转清晰、可视化，易于调试和增加新阶段。
- 极大降低并发竞态和状态不同步导致的前端 Bug 概率。
- 复杂降级逻辑（如主席站点挂掉后选备用主席）在状态机层通过守卫 (Guards) 表达，直观且安全。

**代价 / 妥协：**
- 增加包体积和学习成本（XState v5 的 API 较重）。
- 需要一定量的重构成本以替换掉原有的 Promise 并发控制。
